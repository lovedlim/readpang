1. 왜 PRD가 필요한가

2026년, AI 코딩의 현실

PRD = Product Requirements Document — 만들기 전에 "무엇을, 어떻게" 를 정리한 계획서

AI에게 "독서 모임 앱 만들어줘" → 뭔가 나온다.
하지만 복잡한 서비스를 이렇게 만들면 반드시...

문제실제로 일어나는 일
AI가 임의로 구조를 결정로그인 없이 만들거나, DB 구조가 엉뚱하게 나옴
중간에 방향 수정 불가"벌금 기능 추가해줘" → 기존 코드와 충돌 → 처음부터 다시
예외 처리 누락숫자 대신 문자 입력하면 앱이 멈춤
사용자 상태 미고려모임이 끝났는데도 제출 버튼이 살아있음
디자인이 엉망AI 기본 UI는 밋밋하고, 수정 요청하면 더 이상해짐

단순한 건 그냥 요청해도 된다.
복잡한 서비스는 계획 없이 시작하면 100% 실패한다.
AI는 훌륭한 개발자지만, 기획자도 디자이너도 아니다.

2. 바이브 코딩이란 (2026)

두 가지 핵심 개념

① SDD

Spec-Driven Development

코드 쓰기 전에 명확한 스펙(계획)을 먼저 작성하고, AI가 그 스펙을 바탕으로 코드를 생성

→ PRD 작성이 바로 SDD

② Context Engineering

컨텍스트 엔지니어링

"AI에게 어떤 정보를 얼마나 잘 전달하느냐"
Shopify CEO 토비 뤼트케 + 안드레이 카르파티 제안

→ PRD + 디자인 + Phase별 전달 전부가 이것

SDD

무엇을 만들지 계획 (PRD)

+

Context Eng.

그 계획을 잘 전달 (Phase별 대화)

=

서비스 완성 🚀

2026년 기준 주요 도구

도구단계특징난이도

Google AI Studio ⭐ 개발 대화형 + Build Mode (브라우저에서 앱 생성/배포) 쉬움
Lovable 디자인+개발 프롬프트 → 폴리시된 React 앱 즉시 생성 쉬움
Bolt.new 개발 브라우저에서 웹앱 즉시 생성/배포 쉬움

여기서는 Stitch(디자인) + AI Studio(개발)를 기준으로 합니다.
이유: 둘 다 Google 제품 → 이미지 전달로 연결, 무료, 비개발자에게 가장 접근성 높음

바이브 코딩의 진짜 의미
❌ 기도 코딩

"AI야 앱 만들어줘"

✅ 바이브 코딩

"이 PRD와 이 디자인대로
Step 1부터 만들어줘"

바이브 코딩은
"대충 말하면 알아서 해주는 것"이 아니라,
"명확한 계획과 디자인을 자연어로 전달하는 것"이다.

3. 단순 vs 복잡한 서비스

PRD가 필요한가?

PRD 없이 바로 OK 👍

"오늘 날씨를 보여주는 웹페이지" "할 일 목록 앱" "타이머 앱"

공통점: 사용자 1명, 상태 변화 거의 없음, 데이터 저장 불필요

PRD 반드시 필요 ⚠️

"독서 모임 벌금 관리 서비스" "스터디 출석 체크 서비스" "공동 가계부 서비스"

공통점: 여러 사용자, 역할이 다름, 상태 변화 많음

복잡도 체크리스트 — 3개 이상이면 PRD 필수

□ 로그인이 필요하다

□ 사용자 역할이 2개 이상이다

□ 화면이 3개 이상이다

□ 데이터를 저장하고 불러와야 한다

□ 사용자의 상태가 변한다

□ 시간에 따라 달라지는 규칙이 있다

"읽고팡"은 6개 전부 해당 → PRD 필수

4. 유저플로우 중심 PRD란

왜 "유저플로우 중심"인가

접근법중심 질문비전공자 난이도
기능 중심 "어떤 기능이 필요한가?" 보통
기술 중심 "어떤 API, DB가 필요한가?" 높음
유저플로우 중심 ✅ "사용자가 무엇을 하는가?" 낮음 — 내가 사용자가 되어 상상

"내가 이 앱을 처음 열었다. 그다음 뭘 하지?"

이 질문을 반복하면 된다. 기술 용어는 하나도 몰라도 된다.

❌ 기능 중심 요청

"로그인 기능, 모임 생성 기능,
장 수 입력 기능, 벌금 계산 기능 만들어줘"

→ 로그인 후 빈 화면. 기록이 모임과 연결 안 됨.

✅ 유저플로우 중심

Step 1: 이메일 입력 → 회원가입/로그인 → 홈
Step 2: 모임 만들기 → 6자리 코드 생성 → 카톡 공유
Step 3: 코드 입력 → 최소 장수 설정 → 모임 참여
Step 4: 장 수 입력 → 제출 → 벌금 여부 즉시 표시

→ AI가 각 단계를 순서대로 구현할 수 있다 ✅

PRD 1~4단계 (직접 작성)

내가 가장 잘 아는 것들

1단계

서비스 한 줄 정의

[누가] + [무엇을 해서] + [어떤 결과를 얻는] 서비스

✅ "이메일로 로그인해 독서 모임에 참여하고, 매일 읽은 장 수를 입력해 벌금과 진행 상황을 확인하는 서비스"
2단계

문제 정의

3가지 문장, 사용자가 겪는 고통

  • • 독서 목표는 쉽게 흐트러진다
  • • 매일 기록이 있어야 습관이 된다
  • • 벌금 규칙이 있어야 실행력이 생긴다
3단계

목표 사용자

  • • 독서 습관을 만들고 싶은 사람
  • • 벌금/규칙 기반 동기부여가 필요한 사람
  • • 작은 독서 모임을 운영하고 싶은 사람
4단계

핵심 목표 행동 (딱 1개)

❌ "독서 기록을 하는 것" (모호) ✅ "매일 읽은 장 수를 숫자로 입력하고 제출하는 것"

→ 홈 화면이 결정됨: 숫자 입력 필드 + 제출 버튼

5단계 — 가장 중요 ⭐

핵심 유저플로우 (Happy Path)

각 Step을 사용자 행동 / 시스템 반응 / 저장 정보 3가지로 쓴다

1

이메일 로그인

사용자 행동

이메일/비밀번호 입력 → 로그인 버튼

시스템 반응

홈 화면으로 이동

저장 정보

이메일, 가입 시각, 마지막 접속

2

모임 만들기

사용자 행동

기간/벌금/최소 장수/도서명/인원 입력

시스템 반응

모임 생성 + 6자리 코드 자동 생성

저장 정보

모임 설정값, 생성자, 생성 시각

5

오늘 읽은 장 수 제출 ⭐ 핵심

사용자 행동

숫자 입력 → 제출 버튼

시스템 반응

덮어쓰기 / 미달 시 벌금 표시 / 그래프 업데이트

저장 정보

날짜별 장 수, 수정 시각, 벌금 여부, 누적 벌금

6~9단계 상세

화면 · 상태 · 예외 · 저장

정산

10단계: 데이터 관리 방식

방식설명용도
Firebase ⭐ Google 클라우드 DB.
로그인 내장		 로그인 + 여러 사용자 + 실시간
Supabase 오픈소스 SQL DB 데이터 관계 복잡 + 비용 중요
로컬 저장소 내 브라우저에만 저장 1인용 도구

왜 Firebase인가?

  • ① NoSQL — SQL 없이 자유롭게 저장
  • ② 로그인 기능 내장
  • ③ Google 생태계 (AI Studio 연동 자연스러움)
  • ④ AI가 가장 많이 학습한 백엔드
  • ⑤ 무료 플랜으로 충분 (일 5만 읽기 / 2만 쓰기)
6. AI에게 PRD 작성 도움받기

PRD는 내가 초안, AI는 구체화 파트너

단계내용전략
1~5단계 ⭐서비스 정의 / 문제 / 사용자 / 핵심 행동 / 유저플로우내가 초안 → AI에게 검토·보완
6~10단계화면 / 상태 / 예외 / 저장 / 기술 스택AI 초안 → 내가 수정

AI에게 보내는 프롬프트 (그대로 붙여넣기)

"읽고팡" 완성 PRD 1/2 — 1~5단계

## 1. 서비스 한 줄 정의 이메일로 로그인해 독서 모임에 참여하고, 매일 읽은 장 수를 입력해 벌금과 진행 상황을 확인하는 서비스 ## 2. 문제 정의 - 독서 목표는 쉽게 흐트러진다 - 매일 기록이 있어야 습관이 된다 - 벌금 규칙이 있어야 실행력이 생긴다 ## 3. 목표 사용자 - 독서 습관을 만들고 싶은 사람 - 벌금/규칙 기반 동기부여가 필요한 사람 - 작은 독서 모임을 운영하고 싶은 사람 ## 4. 핵심 목표 행동 매일 읽은 장 수를 숫자로 입력하고 제출하는 것
## 5. 핵심 유저플로우 Step 1: 이메일 로그인 / 회원가입 - 행동: 이메일/비밀번호 입력 → 로그인 버튼 신규 가입 시 닉네임 추가 입력 - 반응: 닉네임 중복 시 "이미 사용 중인 닉네임입니다" 표시 로그인 완료 후 홈으로 이동 - 저장: 이메일, 닉네임, 가입 시각, 마지막 접속 시각 Step 2: 모임 만들기 - 행동: 기간/벌금/최소 장수/도서명/인원 입력 - 반응: 모임 생성, 6자리 코드(영문+숫자) 자동 생성 [코드 복사] 버튼 → "코드가 복사되었습니다" 토스트 - 저장: 모임 설정값, 생성자, 생성 시각, 상태 Step 3: 모임 참여 - 행동: 모임 코드 입력 → 최소 장수 선택 → 참여 - 반응: 참여 완료, 홈에 해당 모임 표시 - 저장: 참여자 목록, 개인 최소 장수, 참여 시각 Step 4: 홈 진입 - 행동: 앱을 연다 - 반응: 참여 중인 모임 1개 + 오늘 상태 표시 - 저장: 마지막 접속 시각 Step 5: 오늘 읽은 장 수 제출 - 행동: 숫자 입력 → 제출 버튼 - 반응: 기존 값 있으면 덮어쓰기 / 미달 시 벌금 표시 / 그래프 즉시 업데이트 - 저장: 날짜별 장수, 수정 시각, 벌금 여부, 누적 벌금

부록: Firebase 보안 규칙 2/2

이 코드를 복사해서 붙여넣으면 된다

rules_version = '2'; service cloud.firestore { match /databases/{database}/documents { // 사용자 정보: 본인만 읽기/쓰기 match /users/{userId} { allow read, write: if request.auth != null && request.auth.uid == userId; } // 모임: 로그인한 사람이면 읽기/만들기 가능 match /groups/{groupId} { allow read: if request.auth != null; allow create: if request.auth != null; allow update: if request.auth != null; allow delete: if false; // 삭제 불가 } // 기록: 본인만 쓰기, 로그인하면 읽기 match /records/{recordId} { allow read: if request.auth != null; allow create, update: if request.auth != null && request.resource.data.userId == request.auth.uid; allow delete: if false; // 삭제 불가 } } }

규칙 읽는 법 (이것만 알면 됨)

allow read: if request.auth != null;

→ 로그인한 사람만 읽을 수 있다

allow write: if request.auth.uid == userId;

→ 본인 데이터만 쓸 수 있다

allow delete: if false;

→ 아무도 삭제할 수 없다

⚠️ 규칙 적용 후 앱 기능이 안 되면?
Chrome → F12 → Console 탭에서 에러 복사
→ AI에게 붙여넣기 → 바로 해결

Q. 로그인 없이 누구나 쓸 수 있는 서비스라면?

⚠️ 일부만 붙여넣으면 오류! 아래 코드 전체를 선택 후 교체해야 합니다

① 읽기 + 등록만 허용

예: 익명 설문, 방명록

rules_version = '2'; service cloud.firestore { match /databases/{db}/documents { match /{document=**} { allow read: if true; allow create: if true; allow update: if false; allow delete: if false; } } }

✅ 남이 내 글을 못 지움

② 수정·삭제도 허용

예: 공개 메모장

rules_version = '2'; service cloud.firestore { match /databases/{db}/documents { match /{document=**} { allow read: if true; allow write: if true; // write = create + // update + delete 포함 } } }

⚠️ 누구나 남의 데이터도 삭제 가능

③ 비밀 토큰으로 본인 확인

예: 링크 공유 수정폼

rules_version = '2'; service cloud.firestore { match /databases/{db}/documents { match /{document=**} { allow read: if true; allow create: if true; allow update, delete: if resource.data.token == request.resource.data.token; } } }

✅ 토큰 아는 사람만 수정

💡 수정·삭제가 필요하면 AI에게 이렇게 요청: "로그인 없이 누구나 읽고 쓸 수 있는 서비스야. 단, 문서를 만든 사람만 수정·삭제할 수 있게 토큰 방식으로 보안 규칙 만들어줘."
